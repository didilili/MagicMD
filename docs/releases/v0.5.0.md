# MagicMD v0.5.0

v0.5.0 是 MagicMD 从“更容易开始使用”继续走向“更适合真实归档和发布维护”的版本。

这次升级重点放在三个方向：输出更完整、回归更可控、发布流程更稳定。MagicMD 仍然专注公开文章链接到 Markdown 内容包的转换，但现在可以额外生成 Word 文档，保留微信公众号封面图，并用更系统的真实样本和发布清单支撑后续迭代。

## 这次升级了什么

- 新增可选 DOCX 导出：`magicmd convert URL --format docx` 或 `[docx] enabled = true` 会在保留 Markdown 内容包的同时生成 `article.docx`。
- DOCX 导出复用最终 `article.md`，由 Pandoc 执行，并支持配置 `pandoc_path` 和 `reference_doc`。
- 修复 `--overwrite` 覆盖同名内容包时旧图片、旧 DOCX 等生成文件可能残留的问题；现在会先清理旧包再写入本次结果。
- 新增微信公众号封面图元数据提取，并默认把封面图写入 Markdown，让归档结果更接近原文章卡片。
- 新增真实站点回归清单和 dry run 记录，覆盖微信公众号、掘金、CSDN 等样本，方便发布前发现解析质量退化。
- 新增手动触发的 PyPI / npm Trusted Publishing workflow；默认可先做 verify dry run，再决定是否发布到 PyPI 和 npm。
- 补齐发布自动化说明和发布后 smoke test 清单，覆盖 GitHub Release、PyPI、npm 和 Agent Skill 安装路径。
- 优化官网首页、Agent Skill 文档和配置生成器体验；配置生成器在宽屏下保留更接近文档页的左右留白。

## 为什么值得升级

v0.1 解决的是“能稳定转文章”。

v0.2 解决的是“输出能适配你的发布系统”。

v0.3 解决的是“业务系统能直接调用 MagicMD”。

v0.4 解决的是“中文用户和 Agent 用户更容易开始使用 MagicMD”。

v0.5 解决的是“归档结果更完整，维护者更有把握发布下一版”。

如果你需要同时得到 Markdown 和 Word 文档，可以这样运行：

```bash
magicmd convert "https://mp.weixin.qq.com/s/example" --format docx
```

如果你希望长期固定输出规则，可以在 `.magicmd.toml` 中启用：

```toml
[docx]
enabled = true
pandoc_path = "pandoc"
reference_doc = ""
```

发布前可以先跑本地检查，再用 GitHub Actions 的 `Publish Release` workflow 做远端 verify：

```text
version: 0.5.0
publish_pypi: false
publish_npm: false
```

# MagicMD v0.5.0

v0.5.0 moves MagicMD from "easier to start with" toward "more reliable for real archiving and release maintenance."

This release focuses on three things: richer output, more controlled regression checks, and a steadier release process. MagicMD still focuses on converting public article URLs into Markdown content packages, but it can now generate Word documents, preserve WeChat cover images, and rely on a clearer live-sample and release-checklist workflow.

## What's Changed

- Added optional DOCX export: `magicmd convert URL --format docx` or `[docx] enabled = true` keeps the Markdown package and also generates `article.docx`.
- DOCX export reuses the final `article.md`, runs through Pandoc, and supports `pandoc_path` plus `reference_doc`.
- Fixed stale generated files when `--overwrite` replaces an existing package; MagicMD now clears the old package before writing the current conversion result.
- Added WeChat cover-image metadata extraction and default Markdown output so archives preserve more of the original article card.
- Added live-site regression manifests and dry-run records for WeChat, Juejin, CSDN, and related samples, making pre-release quality checks more systematic.
- Added a manually triggered PyPI / npm Trusted Publishing workflow. Maintainers can run a verify dry run first, then choose whether to publish to PyPI and npm.
- Added release automation notes and a post-release smoke test checklist covering GitHub Release, PyPI, npm, and Agent Skill installation paths.
- Improved the website homepage, Agent Skill docs, and Config Builder experience; the Config Builder now keeps more document-like side gutters on wide screens.

## Why Upgrade

v0.1 made article conversion reliable.

v0.2 made output configurable for publishing systems.

v0.3 made MagicMD callable from backend applications.

v0.4 made MagicMD easier to start with for Chinese users and agent-driven workflows.

v0.5 makes archived output richer and gives maintainers more confidence before publishing the next release.

To generate both Markdown and Word output:

```bash
magicmd convert "https://mp.weixin.qq.com/s/example" --format docx
```

To make DOCX output part of your long-term config:

```toml
[docx]
enabled = true
pandoc_path = "pandoc"
reference_doc = ""
```

Before publishing, maintainers can run the GitHub Actions `Publish Release` workflow as a remote verify pass:

```text
version: 0.5.0
publish_pypi: false
publish_npm: false
```
