# MagicMD v0.6.0

v0.6.0 解决的是：文章已经转换成 Markdown 内容包之后，怎么安全地发布到 GitHub 内容仓库。

以前 MagicMD 主要负责生成 `output/` 内容包；从这一版开始，它也可以按配置创建发布分支、提交文件，并可选创建 Pull Request。你可以先用 `--dry-run` 看清楚会写入哪些文件，再决定是否真正推送。

这条线也延续了之前几个版本的节奏：

- v0.1：稳定转换公开文章。
- v0.2：让输出适配不同发布系统。
- v0.3：提供 Python SDK，方便业务系统直接调用。
- v0.4：补齐中文优先、Agent 友好的上手体验。
- v0.5：让归档结果更完整，发布维护更稳。
- v0.6：把转换结果发布到 GitHub 内容仓库。

## 这版新增了什么

- 新增 `magicmd publish github URL` 命令，把转换结果发布到指定 GitHub 内容仓库。
- 支持从 `.magicmd.toml` 读取 `[publish.github]` 配置，适合长期发布到同一个内容仓库。
- 支持把 `GITHUB_TOKEN` 放在本地 `.env` 中，避免每次手动 `export`。
- 支持 `--dry-run` 预览发布计划，不创建分支、不提交、不推送、不创建 PR。
- 支持配置目标仓库、目标目录、发布分支、提交信息、是否创建 PR、是否覆盖已有文件。
- 配置生成器新增 GitHub 发布配置区域，新用户可以先生成配置，再执行发布命令。

## 推荐用法：先配置，再 dry-run，再发布

第一次使用时，先准备一个内容仓库，例如 `owner/content`，然后在项目根目录放一份 `.magicmd.toml`：

```toml
[publish.github]
repo = "owner/content"
target_dir = "content/posts/{date}-{slug}"
branch = "magicmd/{slug}"
commit_message = "发布文章：{title}"
create_pr = true
overwrite = false
```

再把 GitHub token 放进本地 `.env`：

```env
GITHUB_TOKEN=ghp_xxx
```

确认配置后，先 dry-run：

```bash
magicmd publish github "https://mp.weixin.qq.com/s/example" --dry-run
```

输出正常后，再真实发布：

```bash
magicmd publish github "https://mp.weixin.qq.com/s/example"
```

## 发布到 GitHub 时会发生什么

`magicmd publish github` 会先转换文章，再根据 `[publish.github]` 生成发布计划。

如果不是 dry-run，它会创建临时 worktree，切到配置里的发布分支，把 `article.md`、`metadata.json`、`extraction-report.json` 和媒体文件写入目标目录，然后提交并推送到内容仓库。

如果配置了 `create_pr = true`，MagicMD 会在推送分支后创建 Pull Request。也就是说，默认流程不是直接改你的主分支，而是让你在 GitHub 上 review 之后再合并。

## 稳定性与安全

- 发布前会检查提取质量，避免把失败页面、URL 标题或调试文件误推到内容仓库。
- GitHub 发布流程使用临时 worktree，不会把 token 写入 git remote URL。
- 已加固已有 PR 复用、分支名校验、目标路径校验和目标目录文件处理。
- 这个版本没有要求用户迁移旧配置；原有转换命令继续可用。

# MagicMD v0.6.0

v0.6.0 adds a GitHub publishing workflow for converted Markdown packages.

MagicMD used to stop at generating an `output/` package. This release lets you publish that package to a GitHub content repository through a configurable branch-and-PR workflow. You can preview the full plan with `--dry-run` before any remote write happens.

## What Changed

- Added `magicmd publish github URL`.
- Added `[publish.github]` support in `.magicmd.toml`.
- Added local `.env` loading for `GITHUB_TOKEN`.
- Added `--dry-run` for publishing previews.
- Added configuration for repository, target directory, branch name, commit message, PR creation, and overwrite behavior.
- Added GitHub publishing options to the Config Builder.

## Recommended Flow

Create a content repository, configure `.magicmd.toml`, put `GITHUB_TOKEN` in local `.env`, then run:

```bash
magicmd publish github "https://mp.weixin.qq.com/s/example" --dry-run
magicmd publish github "https://mp.weixin.qq.com/s/example"
```

When `create_pr = true`, MagicMD pushes a publishing branch and opens a Pull Request instead of writing directly to your main branch.
