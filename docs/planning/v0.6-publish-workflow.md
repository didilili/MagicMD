# v0.6 Publish Workflow Design

v0.5 已经把 MagicMD 推到一个比较完整的产品阶段：用户可以安装、转换公开文章、生成内容包、配置 Markdown 输出、使用 SDK、通过 Agent Skill 调用，并通过 PyPI/npm/GitHub Release 获取稳定版本。

v0.6 的重点不是继续扩大平台适配，而是补上用户转换后的下一步：把内容包发布到自己的 GitHub 内容仓库。

## Product Positioning

MagicMD 仍然是一个轻量、可组合的文章内容入库工具：

```text
公开文章链接 -> Markdown 内容包 -> 可选发布到内容仓库
```

发布工作流必须保持可选，不影响现有的核心转换能力。转换仍然由 `convert_article()` 和现有 CLI 完成；发布层只消费转换结果和输出目录，不直接参与平台检测、抓取、解析或 Markdown 渲染。

## User Stories

### Story 1: 先看发布计划

用户想确认一篇文章会被写到哪个仓库、哪个目录、哪个分支，以及会提交哪些文件。

```bash
magicmd publish github "https://mp.weixin.qq.com/s/example" \
  --repo owner/content \
  --target-dir content/posts \
  --branch magicmd/{slug} \
  --dry-run
```

MagicMD 应该展示：

- 文章标题、平台、来源 URL。
- 本次转换生成的内容包路径。
- 目标仓库、目标分支、目标目录。
- 将要写入的文件列表。
- commit message。
- 是否会创建 Pull Request。

`--dry-run` 不得写入目标仓库、不得创建分支、不得 push、不得创建 PR。

### Story 2: 发布到 GitHub 分支

用户确认 dry-run 结果后，执行实际发布。

```bash
magicmd publish github "https://mp.weixin.qq.com/s/example" \
  --repo owner/content \
  --target-dir content/posts \
  --branch magicmd/{slug}
```

MagicMD 应该：

- 复用 SDK 完成文章转换。
- 把生成内容包中的 Markdown、metadata、report、图片、视频、DOCX 等文件复制到目标目录。
- 创建或更新目标分支。
- 提交 commit。
- push 到远端。
- 输出最终分支和提交信息。

### Story 3: 发布后创建 Pull Request

用户希望把内容导入流程接入已有内容仓库的 review 流程。

```bash
magicmd publish github "https://mp.weixin.qq.com/s/example" \
  --repo owner/content \
  --target-dir content/posts \
  --branch magicmd/{slug} \
  --pr
```

MagicMD 应该在 push 成功后创建 PR，并输出 PR 链接。第一版只需要支持从发布分支到仓库默认分支创建 PR。

## Recommended CLI Shape

```bash
magicmd publish github URL [OPTIONS]
```

Required options:

- `--repo owner/name`: GitHub 仓库。
- `--target-dir PATH`: 目标仓库中的内容目录。

Common options:

- `--branch TEMPLATE`: 发布分支模板，默认 `magicmd/{slug}`。
- `--commit-message TEMPLATE`: commit message 模板，默认 `Add article: {title}`。
- `--config PATH`: 复用现有 `.magicmd.toml`。
- `--output PATH`: 本地转换输出目录，默认沿用现有输出配置。
- `--dry-run`: 只展示计划，不写远端。
- `--pr`: push 后创建 Pull Request。
- `--overwrite`: 允许覆盖目标目录中同名文件。

Template variables should reuse existing MagicMD variables where possible:

- `{title}`
- `{slug}`
- `{date}`
- `{published_at}`
- `{author}`
- `{platform}`
- `{short_hash}`
- `{content_hash}`

## Configuration Shape

`.magicmd.toml` 可以新增可选发布配置：

```toml
[publish.github]
repo = "owner/content"
target_dir = "content/posts"
branch = "magicmd/{slug}"
commit_message = "Add article: {title}"
create_pr = false
overwrite = false
```

CLI 显式参数优先级高于配置文件。

## Architecture

v0.6 应该新增一个独立 publishing 模块，避免污染转换核心。

Suggested modules:

```text
src/magicmd/publish/
├── __init__.py
├── models.py
├── planner.py
├── github.py
└── errors.py
```

Responsibilities:

- `models.py`: 发布配置、发布计划、发布结果的数据模型。
- `planner.py`: 根据 `ArticleConversionResult`、输出目录和配置生成 dry-run 计划。
- `github.py`: 执行 GitHub 仓库写入、分支、commit、push、PR 创建。
- `errors.py`: 发布相关错误，例如权限缺失、目标冲突、Git 命令失败、PR 创建失败。

Existing modules should stay focused:

- `sdk.py`: 继续负责文章转换。
- `output.py`: 继续负责本地内容包写入。
- `cli.py`: 只增加命令入口，把转换结果交给 publishing 模块。

## Data Flow

```text
URL
  -> convert_article()
  -> ArticleConversionResult
  -> publish planner
  -> PublishPlan
  -> dry-run output or GitHub publisher
  -> PublishResult
```

`PublishPlan` 是关键边界。dry-run 和真实发布都应该依赖同一份计划，避免展示结果和实际写入行为不一致。

## Authentication

第一版推荐使用环境变量：

```bash
export GITHUB_TOKEN=ghp_xxx
```

Token 权限要求：

- 对目标仓库有 contents write 权限。
- 使用 `--pr` 时需要 pull requests write 权限。

如果缺少 token，MagicMD 应该在进入远端写入前失败，并提示：

- 当前缺少 `GITHUB_TOKEN`。
- dry-run 不需要 token。
- 真实发布需要 token 和对应仓库权限。

## Conflict Handling

默认策略是保守的：

- 目标目录不存在：创建。
- 目标目录存在且为空：写入。
- 目标目录存在且有文件：失败并提示使用 `--overwrite`。
- `--overwrite` 只覆盖本次计划内的目标文件，不删除无关文件。

这样可以避免用户误删已有内容仓库文件。

## Error Handling

Publishing errors should be actionable and not hide conversion output.

Examples:

- 转换失败：沿用现有转换错误。
- 目标文件冲突：显示冲突文件路径和 `--overwrite` 提示。
- token 缺失：提示设置 `GITHUB_TOKEN`。
- push 失败：保留本地转换输出，显示失败阶段。
- PR 创建失败：保留已经 push 的分支，输出分支 URL 和失败原因。

## Non-goals

v0.6 不做这些事情：

- 不新增登录站点、付费内容、验证码内容支持。
- 不把 MagicMD 改造成通用爬虫平台。
- 不做网页 UI。
- 不做 HaoGit 专用导入逻辑。
- 不直接写数据库或 CMS API。
- 不同时发布到多个平台。
- 不在 Agent Skill 里复制发布逻辑；Agent Skill 仍然调用 CLI。

## Testing Strategy

Unit tests:

- 发布配置解析。
- branch 和 commit message 模板渲染。
- dry-run plan 生成。
- 目标路径规范化。
- 文件冲突检测。
- token 缺失错误。

Integration-style tests:

- 使用临时本地 git 仓库模拟目标仓库。
- 验证 dry-run 不写入文件。
- 验证真实发布会创建目标目录、复制文件并生成 commit。
- mock GitHub PR API，验证 `--pr` 的请求参数和失败提示。

CLI tests:

- `magicmd publish github URL --dry-run` 输出计划。
- CLI 参数覆盖 `.magicmd.toml` 中的发布配置。
- 缺少 `--repo` 或 `--target-dir` 时给出清晰错误。

## Success Criteria

v0.6 完成时，用户应该能完成这条路径：

```text
安装 MagicMD
  -> 转换一篇公开文章
  -> dry-run 查看发布计划
  -> 发布到 GitHub 内容仓库分支
  -> 可选创建 PR
```

维护者应该能确认：

- 原有 `convert`、`batch`、SDK、Agent Skill 不受影响。
- 发布层有清晰边界。
- dry-run 和真实发布共用同一份计划。
- 没有 token 时仍然可以安全执行 dry-run。
- 发布失败不会破坏已生成的本地内容包。

## Recommended Implementation Order

1. 增加发布配置和数据模型。
2. 实现 `PublishPlan` 和 dry-run 输出。
3. 增加 CLI 入口，但第一阶段只支持 dry-run。
4. 增加本地 git 仓库发布能力。
5. 增加 GitHub token push。
6. 增加可选 PR 创建。
7. 更新官网和快速开始文档。

这个顺序可以让 v0.6 每一步都有可验证结果，避免一开始就把转换、git、GitHub API 和 PR 都搅在一起。
